#输出映射

resultType
resultType中的sql语句中的字段必须与pojo里面的字段相同，如果不相同，那么查询出来的结果只是给你的属性赋初值。
resultType的返回类型决定了mybatis会调用那么查询函数(selectOne,selectList)

resultMap
mybatis中使用resultMap返回类型来完成高级映射(一对一，一对多，多对一)
eg:
还是上面的列子
select ParameterType=”User” resultMap=””>
slect id _id ,name _name from emp ;

pojo
public class User{
private int id;
private String name;
}
这里可以看出，pojo里面的属性和sql里面的字段是不同的，如果使用resultType的话，那么查询查来的id,name就是0，null ,显然这样是错误的，并不是我们要的结果。因为sql片段中的字段和pojo里面的属性不同

activiMQ

By Jackey

發表於 2017-11-26

消息中间件activiMQ
为什么消息中间件会出现？
CORBA,DCOM.RMI和RPC中间件已经广泛的使用在各个领域，但是，面对规模和复杂度越来越高的分布式，这些技术也显示出了其局限性。
面向消息中间件(Messsage Oriented Middleware)较好的解决了以上的问题，发送者将若干的消息发送给消息服务器，消息服务器将消息存放在若干的队列之中，在合适的时候将消息转发给接收者。
重要 ：这种时候，发送和接收是异步的，发送者不必等待，两者的生命周期未必相同（因为消息发送者发送的消息存放在队列之中，发送者发送完消息，它死了，但是，它发送的消息依然存在），发送消息的时候，接收者不一定运行，接收消息的时候，发送消息者不一定运行。

GOD23-设计模式-行为型-观察者模式
观察者模式也被称为：发布-订阅模式，模型-视图模式，源-收听者模式
定义：观察者模式定义了对象间一对多的关系，以便一个对象发生改变的时候，其他的多个对象也要随着更新。

核心
subject ；抽象类，定义了发布者可以增加，删除，提醒观察者更新的方法
concreateSubject ； 继承抽象类，只有设置状态的字段
observer :接口类，只有更新的方法
observerA,observerB :实现了观察者类接口，可以有多个
