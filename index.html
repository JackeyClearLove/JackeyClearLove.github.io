#输出映射

resultType
resultType中的sql语句中的字段必须与pojo里面的字段相同，如果不相同，那么查询出来的结果只是给你的属性赋初值。
resultType的返回类型决定了mybatis会调用那么查询函数(selectOne,selectList)

resultMap
mybatis中使用resultMap返回类型来完成高级映射(一对一，一对多，多对一)
eg:
还是上面的列子
select ParameterType=”User” resultMap=””>
slect id _id ,name _name from emp ;

pojo
public class User{
private int id;
private String name;
}
这里可以看出，pojo里面的属性和sql里面的字段是不同的，如果使用resultType的话，那么查询查来的id,name就是0，null ,显然这样是错误的，并不是我们要的结果。因为sql片段中的字段和pojo里面的属性不同

activiMQ

By Jackey

发表于 2017-11-26

消息中间件activiMQ
为什么消息中间件会出现？
CORBA,DCOM.RMI和RPC中间件已经广泛的使用在各个领域，但是，面对规模和复杂度越来越高的分布式，这些技术也显示出了其局限性。
面向消息中间件(Messsage Oriented Middleware)较好的解决了以上的问题，发送者将若干的消息发送给消息服务器，消息服务器将消息存放在若干的队列之中，在合适的时候将消息转发给接收者。
重要 ：这种时候，发送和接收是异步的，发送者不必等待，两者的生命周期未必相同（因为消息发送者发送的消息存放在队列之中，发送者发送完消息，它死了，但是，它发送的消息依然存在），发送消息的时候，接收者不一定运行，接收消息的时候，发送消息者不一定运行。

GOD23-设计模式-行为型-观察者模式
观察者模式也被称为：发布-订阅模式，模型-视图模式，源-收听者模式
定义：观察者模式定义了对象间一对多的关系，以便一个对象发生改变的时候，其他的多个对象也要随着更新。

核心
subject ；抽象类，定义了发布者可以增加，删除，提醒观察者更新的方法
concreateSubject ； 继承抽象类，只有设置状态的字段
observer :接口类，只有更新的方法
observerA,observerB :实现了观察者类接口，可以有多个

GOF23-设计模式-行为型-策略模式
定义 ： 解决某一个问题的一个算法族，允许用户从该算法族中任选一个算法解决某一问题，同时可以方便的更换算法和添加新的算法，并且有客户端来决定使用哪个算法

案列 ：超市针对不同的客户，优惠不同，比如：老客户大批量购买（7折），新客户大批量购买（8.5浙），老客户小批量购买（8.8浙），新客户小批量购买（不打折）
在这个案例之中，解决的是问题是：得到各种不同客户的最终价格，算法族是：不同的客户，打折的策略不同

GOF23-decorator

By Jackey

发表于 2017-11-25

GOF23-设计模式-结构型模式-装饰者模式
定义：动态的为对象添加新的功能
装饰者模式是一种用于代替继承的技术。无须通过继承增加子类就能扩展对象的新功能，使用对象的关联关系来代替继承关系，更加灵活，同时避免类型体系的快速膨胀。
比如：一面白色的墙，上面粘了挂钩，它就拥有了挂东西的功能，墙上安装了风扇，它就拥有了提供吹风的功能。 墙就相当于初始的对象，而挂钩，风扇就相当于给墙装饰的新功能。

GOF23-设计模式-结构型模式-组合模式
组合模式和java基础里面的组合是不同的
组合是：A类里面定义了B类，那么A类就拥有了B类的属性方法
组合模式：把部分和整体的关系用树形来表示结构表示，从而使客户端可以统一的处理部分对象和整体对象。
组合的核心 ： 抽象构件（component）角色：定义了叶子和容器构件的共同点，
叶子构件角色：无子节点
容器构件角色：有容器特征，可以包含子节点

GOF23-结构型模式-桥接模式
一般用于商品分类
核心：用于处理多层继承结构，处理多维度变化的场景，将各个维度设计成独立的继承结构，是各个维度可以独立的扩展而不影响其他维度

案例
台式电脑，笔记本电脑，联想台式电脑，神州台式电脑，联想笔记本电脑，神州笔记本电脑
现在就可以建立一个二维，x轴指的是商品分类，y轴指品牌分类，然后用一个桥梁把x,y轴连接起来，并且x,y增加产品可以随意的增加，他们也不会相互影响。
